# Voice Assistant Bridge V2 开发计划（按步执行）

- 计划版本：`v0.1`
- 更新时间：`2026-03-14`
- 关联设计：`DETAILED_DESIGN_V2.md`（v0.3）
- 目标：按“方案 A 主线 + B/C 备案”完成会议模式端到端可用版本

## 1. 已冻结前提

1. 主线方案：会中沿用当前实时 API，会后 Windows 加精（`faster-whisper`）。
2. 会议模式：人工开关，进入后安卓本地可靠录音优先。
3. 会议结束：自动触发加精，后台异步处理，不阻塞主体功能。
4. 音频保留：安卓本地保留 7 天，前提是已成功上传 Windows 备份。
5. 图片：原图上传并传入 OpenClaw。
6. 唤醒词：单录音链路复用，不新增第二条麦克风采集链路。

## 2. 交付里程碑

### 2.1 M1 基础底座（Week 1）

目标：打通会议模式最小闭环，不做加精。

任务：

1. 后端：新增会议主表与事件表（`meeting_sessions`、`meeting_events`）。
2. 后端：新增 `POST /v2/meetings`、`POST /v2/meetings/{id}/mode`、`GET /v2/events/stream`。
3. Android：新增“会议模式开关”UI 与状态提示。
4. Android：新增录音分片落盘（30s 分片 + manifest）。
5. Android：新增 PCM 分发总线（落盘/STT/KWS 共享）。
6. Android：新增唤醒词状态机（监听/命令窗/冷却）骨架。
7. Windows UI：新增会议控制台壳层（开始/结束/当前状态）。

验收：

1. 能开始/结束会议。
2. 录音分片在安卓本地可见、可回放、可索引。
3. 事件流能看到 `meeting.mode_on/off`、`audio.segment.*`。
4. 唤醒词启用时无麦克风冲突（不出现双录音链启动失败）。

### 2.2 M2 备份与历史（Week 2）

目标：会后音频可回传、可校验、可历史化。

任务：

1. 后端：新增 `POST /v2/meetings/{id}/audio:upload`。
2. 后端：实现 checksum 校验与上传状态机（pending/uploaded/failed）。
3. 后端：新增 `audio_segments` 表与 `upload_manifest` 落盘。
4. Windows UI：历史会议列表、音频备份状态面板。
5. Android：上传重试队列（指数退避 + 幂等分片 ID）。
6. Android：唤醒词触发事件上报（`wakeword.*`）。

验收：

1. 会议结束后音频可自动上传至 Windows 仓库。
2. 上传失败可重试，最终状态可追踪。
3. 历史会议卡片可查看备份完整性。
4. 唤醒词触发可在 Windows 事件流中观测。

### 2.3 M3 自动加精（Week 3）

目标：会后自动加精转写上线，且后台异步不阻塞。

任务：

1. 后端：新增 `transcription_jobs` 表和任务状态机（queued/running/success/failed）。
2. 后端：新增后台 worker 线程池（`ThreadPoolExecutor` 或独立 asyncio 任务队列）。
3. 后端：会议结束自动创建加精任务。
4. 后端：接入 `faster-whisper` 会后批处理。
5. 后端：新增 `refined.jsonl` 与版本号机制（`revision`）。
6. Windows UI：任务面板（排队/运行/完成/失败 + 日志）。
7. Android：TTS 播报期间抑制唤醒词，避免自唤醒。

验收：

1. 会议结束后自动触发加精。
2. 主接口响应不被长任务阻塞（P95 响应保持稳定）。
3. 加精完成后稳定稿可检索、可展示。
4. TTS 播报期间唤醒词误触发率显著下降（按测试阈值验收）。

### 2.4 M4 说话人细分 V1（Week 4）

目标：会后细粒度说话人区分上线（弱识别）。

任务：

1. 后端：新增 `meeting_segments_refined` 表字段（`speaker_cluster_id/confidence/name/source`）。
2. 后端：接入 diarization 任务步骤（优先与加精任务串联）。
3. Windows UI：说话人手工改名与回写。
4. 后端：改名审计记录（保留历史映射）。

验收：

1. 会后稳定稿带说话人簇 ID。
2. 支持人工改名并在历史会次复用。

### 2.5 M5 图片原图链路（Week 5）

目标：原图上传 -> OpenClaw 分析 -> 时间轴回链。

任务：

1. 后端：新增 `POST /v2/meetings/{id}/images:upload`。
2. Android：拍照入口、原图上传、上传状态展示。
3. 后端：保存原图与元数据（拍摄时间、设备、尺寸、hash）。
4. 后端：调用 OpenClaw 图片分析（原图输入）。
5. Windows UI：图片与时间轴联动展示。

验收：

1. 原图不降分辨率入库。
2. OpenClaw 分析结果可回看并关联会议片段。

### 2.6 M6 报告与发布（Week 6）

目标：输出可交付版本与运行手册。

任务：

1. 三层报告（brief/action/deep）模板与生成器。
2. 性能与稳定性压测（长会、弱网、断点恢复）。
3. 数据保留策略落地（7 天清理守卫）。
4. 发布脚本与回滚方案。

验收：

1. 连续 60 分钟会议可稳定跑通。
2. 断网恢复后数据不丢。
3. 7 天自动清理只清“已上传成功”数据。

## 3. 任务依赖图（执行顺序）

1. M1 -> M2（无 M1 事件/会话基础则无法做备份）
2. M2 -> M3（无备份与历史，自动加精无输入）
3. M3 -> M4（说话人细分依赖稳定稿）
4. M2 与 M5 可并行（音频链路与图片链路互不阻塞）
5. M3/M4/M5 -> M6（统一验收发布）
6. KWS 状态机在 M1 落地，M2 完成事件贯通，M3 完成抑制策略。

## 4. 技术实现细则（关键约束）

### 4.1 异步加精不阻塞

1. `/v2/meetings/{id}/mode` 结束会议接口只入队任务，不同步等待完成。
2. Worker 独立线程池执行 STT/diarization。
3. 任务进度通过 `transcription_jobs` + WebSocket 事件回传 UI。

### 4.2 7 天清理守卫

1. 清理 job 仅扫描 `uploaded=true AND uploaded_at <= now-7d`。
2. 先写删除审计日志，再物理删除文件。
3. 删除失败自动重试，最多 3 次后告警。

### 4.3 备用方案归档

1. 在 `docs/tech-options/` 维护 B/C 方案接入说明和验证结论。
2. 主线代码不引入 B/C 运行时依赖，避免首期复杂化。

### 4.4 唤醒词不冲突约束

1. Android 全局只允许一个 `AudioRecord` 持有者（会议录音器）。
2. KWS/STT/落盘通过 PCM 分发总线复用，不允许各自创建录音器。
3. 命令窗与冷却期由状态机管控，避免重复唤醒和链路重入。

## 5. 测试计划

### 5.1 自动化测试

1. API 合约测试（v2 会议、上传、任务状态）。
2. 存储一致性测试（事件序、分片校验、任务幂等）。
3. Worker 并发测试（队列积压、失败重试、取消任务）。

### 5.2 场景测试

1. 30/60/120 分钟会议稳定性。
2. 网络断连 1~10 分钟后补传。
3. 会议结束立刻开始下一场会议（并发会次隔离）。
4. 安卓空间不足告警与保护行为。
5. 连续播报（TTS）+ 现场人声混合场景下唤醒词误触发测试。

## 6. 风险与应对

1. 风险：会后加精任务耗时长。  
   应对：任务队列 + 并发上限 + 优先级控制。
2. 风险：安卓端存储压力。  
   应对：剩余空间阈值告警 + 上传优先 + 清理守卫。
3. 风险：会中中英混说质量波动。  
   应对：会后加精兜底 + 术语字典增量。

## 7. 执行方式（按步推进）

1. 先实现 M1 并过验收，再进入 M2。
2. 每个里程碑结束输出：变更说明、测试结果、风险清单。
3. 任一里程碑出现 P0 阻塞，暂停后续开发先清障。
